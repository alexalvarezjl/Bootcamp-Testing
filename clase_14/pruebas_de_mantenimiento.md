# Pruebas de Mantenimiento

Las pruebas de mantenimiento son fundamentales para garantizar que el software siga funcionando correctamente a lo largo del tiempo, incluso después de realizar cambios, migraciones o bajas parciales. Algunos tipos de pruebas de mantenimiento comunes incluyen:

1. **Pruebas por cambios:** Cuando se realizan modificaciones en el software, ya sea para corregir errores o para agregar nuevas funcionalidades, es importante realizar pruebas para asegurarse de que los cambios no afecten negativamente al sistema existente.

2. **Pruebas por migraciones:** Si se realiza una migración del software a una nueva plataforma, sistema operativo o base de datos, es esencial llevar a cabo pruebas exhaustivas para garantizar que la migración se haya realizado correctamente y que el software funcione correctamente en el nuevo entorno.

3. **Pruebas por bajas parciales:** En algunos casos, es posible que se realicen bajas parciales de software o sistemas, donde se eliminan o desactivan ciertas partes del software. En estos casos, se deben realizar pruebas para asegurarse de que las áreas restantes del software sigan funcionando correctamente y no se vean afectadas por la baja parcial.

En general, las pruebas de mantenimiento tienen como objetivo garantizar la estabilidad y la calidad del software a lo largo del tiempo, incluso después de realizar cambios o modificaciones. Esto ayuda a mantener la satisfacción del cliente y a prevenir problemas futuros.

## Mantenimiento debido a un cambio

El mantenimiento debido a un cambio puede involucrar diferentes tipos de modificaciones en el software. Algunos de los tipos comunes de mantenimiento debido a un cambio son:

1. Perfectivo - Cambios Planeados (Nuevas Funcionalidades): Estos cambios implican la adición de nuevas funcionalidades o mejoras al software existente. Pueden surgir a partir de nuevos requisitos o solicitudes del cliente para agregar más capacidades al sistema. Las pruebas de mantenimiento en este caso se centran en asegurar que las nuevas funcionalidades se implementen correctamente y no introduzcan errores en el sistema existente.

2. Correctivas: Estos cambios se realizan para corregir errores o defectos identificados en el software. Las pruebas de mantenimiento en este caso se enfocan en verificar que los errores se hayan corregido de manera efectiva y que no se hayan introducido nuevas fallas en el proceso de corrección.

3. Cambios de Emergencia: Estos cambios son necesarios cuando surgen situaciones críticas que requieren una respuesta rápida para solucionar problemas graves en el software. Las pruebas de mantenimiento en este caso se enfocan en validar la solución de emergencia y asegurar que el software siga funcionando correctamente.

4. Adaptativo - Cambios de Ambiente: Estos cambios se realizan cuando es necesario adaptar el software a un nuevo entorno o contexto, como cambios en la infraestructura tecnológica o en los requisitos de seguridad. Las pruebas de mantenimiento en este caso se centran en verificar que el software funcione correctamente en el nuevo ambiente y que no se hayan introducido problemas de compatibilidad.

En resumen, el mantenimiento debido a un cambio implica diferentes tipos de modificaciones en el software, como la adición de nuevas funcionalidades, la corrección de errores, la solución de problemas urgentes y la adaptación a nuevos entornos. Las pruebas de mantenimiento son esenciales para garantizar que los cambios se implementen correctamente y que el software siga funcionando de manera óptima.

## Mantenimiento debido a una migración

El mantenimiento debido a una migración implica realizar cambios en el software para adaptarlo a una nueva plataforma o sistema operativo. Algunos de los casos comunes de mantenimiento debido a una migración son:

1. Cambios por Migración: Estos cambios se realizan cuando se migra el software de una plataforma a otra. Por ejemplo, migrar de un servidor local a la nube o cambiar de un sistema operativo a otro. Las pruebas de mantenimiento en este caso se centran en verificar que el software funcione correctamente en la nueva plataforma y que no se hayan introducido errores durante el proceso de migración.

2. Cambios por Cambio de Plataforma: Estos cambios ocurren cuando se cambia de una plataforma tecnológica a otra. Por ejemplo, cambiar de un framework de desarrollo a otro o migrar de una base de datos a otra. Las pruebas de mantenimiento se enfocan en validar que el software funcione adecuadamente en la nueva plataforma y que no haya incompatibilidades o problemas de rendimiento.

3. Cambios por Cambio de Sistema Operativo: Estos cambios se realizan cuando se migra el software a un nuevo sistema operativo. Por ejemplo, pasar de Windows a Linux o de macOS a Windows. Las pruebas de mantenimiento se centran en asegurar que el software sea compatible con el nuevo sistema operativo y que funcione correctamente en él.

Durante el mantenimiento debido a una migración, es importante realizar pruebas exhaustivas para garantizar que el software se adapte sin problemas a la nueva plataforma o sistema operativo. Esto incluye verificar la funcionalidad, la compatibilidad, el rendimiento y la estabilidad del software en el nuevo entorno.

## Mantenimiento debido a bajas

El mantenimiento debido a bajas se refiere a los cambios realizados en el software cuando se eliminan funcionalidades o se deja de utilizar un sistema. Algunos casos comunes de mantenimiento debido a bajas son:

1. Eliminación de funcionalidad: En ocasiones, es necesario eliminar ciertas funcionalidades del software debido a cambios en los requerimientos o decisiones estratégicas. El mantenimiento en este caso implica identificar y eliminar el código, configuraciones y recursos relacionados con la funcionalidad que ya no se utilizará.

2. Migración de información de otro sistema: Cuando se migra información de un sistema a otro, puede requerir cambios en el software existente para adaptarlo y asegurar una correcta integración. Esto implica ajustes en la estructura de datos, interfaces de comunicación y procesos de migración de datos.

3. Desactivación de un sistema en convivencia: En situaciones en las que dos sistemas han coexistido durante un largo período y uno de ellos se deja de utilizar, se requiere un mantenimiento para desactivar y eliminar el sistema obsoleto. Esto incluye la migración de datos, la actualización de enlaces o integraciones, y la limpieza general del entorno.

Durante el mantenimiento debido a bajas, es importante realizar pruebas exhaustivas para verificar que la funcionalidad eliminada o desactivada no afecte negativamente el funcionamiento del software. Además, se deben tomar medidas para asegurar una transición suave y minimizar cualquier impacto en los usuarios o en el sistema en general.

## Pruebas de Mantenimiento

Las pruebas de mantenimiento son esenciales para garantizar la calidad y la estabilidad del software después de realizar cambios. Estas pruebas no solo deben evaluar el cambio en sí, sino también las partes del software que no han cambiado. El alcance de las pruebas de mantenimiento está determinado por los siguientes factores:

- Riesgos del cambio: El alcance de las pruebas debe considerar los posibles riesgos asociados con el cambio realizado. Esto implica identificar las áreas del software que podrían verse afectadas indirectamente debido al cambio y asegurarse de que se realicen pruebas adecuadas en esas áreas.

- Tamaño del sistema existente: El alcance de las pruebas también está relacionado con el tamaño del sistema existente. En sistemas grandes y complejos, es necesario realizar pruebas exhaustivas en diferentes componentes y funcionalidades para garantizar que no se produzcan efectos secundarios no deseados.

- Tamaño del cambio: El tamaño del cambio realizado también influye en el alcance de las pruebas de mantenimiento. Si el cambio es menor y tiene un impacto limitado en el sistema, las pruebas pueden centrarse en áreas específicas relacionadas directamente con el cambio. Sin embargo, si el cambio es significativo, se requerirán pruebas más exhaustivas en múltiples áreas del software.

Es importante tener en cuenta que las pruebas de mantenimiento deben ser planificadas y documentadas adecuadamente. Esto incluye identificar los casos de prueba relevantes, definir los criterios de aceptación y registrar los resultados de las pruebas. Además, es esencial contar con un entorno de pruebas representativo que refleje las condiciones del sistema en producción.

Las pruebas de mantenimiento pueden variar según el tipo de cambio realizado en el software. En general, estas pruebas deben abarcar todos los niveles de testing y pueden involucrar cualquier o todos los tipos de pruebas disponibles. 

La determinación de cómo el sistema existente es afectado por los cambios, a través del análisis de impacto, es crucial para decidir la cantidad de pruebas de regresión que se deben realizar. El análisis de impacto ayuda a identificar las áreas del sistema que podrían verse afectadas indirectamente debido al cambio y permite enfocar los esfuerzos de prueba en esas áreas críticas.

**Dependiendo de la naturaleza del cambio, las pruebas de mantenimiento pueden incluir:**

- Pruebas unitarias: Se realizan para verificar el correcto funcionamiento de las unidades de código modificadas y asegurarse de que no se hayan introducido errores.

- Pruebas de integración: Se llevan a cabo para probar la interoperabilidad y la correcta integración de los componentes modificados con el resto del sistema.

- Pruebas de sistema: Se realizan para evaluar el comportamiento general del sistema después de los cambios y asegurarse de que todas las funcionalidades principales sigan funcionando correctamente.

- Pruebas de aceptación: Se llevan a cabo para verificar que los cambios cumplan con los requisitos y expectativas establecidos por el cliente o el usuario final.

Es importante tener en cuenta que las pruebas de mantenimiento deben ser planificadas y ejecutadas de manera adecuada, siguiendo un enfoque sistemático y documentando los resultados obtenidos. Esto garantiza la calidad y estabilidad del software después de realizar los cambios y ayuda a mitigar los posibles riesgos asociados con el mantenimiento del sistema.

## Tipos de Regresiones

Existen diferentes tipos de regresiones que se pueden realizar como parte de las pruebas de software, y dos de ellos están relacionados con los cambios:

1. Test de Confirmación:
   - Este tipo de prueba se realiza después de que se ha solucionado un defecto o se ha completado la prueba de una nueva funcionalidad. El objetivo principal es confirmar que el problema reportado se ha solucionado correctamente o que la nueva funcionalidad se ha implementado de manera satisfactoria. En esta prueba, se enfoca en el área afectada por el cambio y se verifica que funcione como se espera.

2. Test de Regresión:
   - El test de regresión se realiza para asegurar que los cambios realizados en el software no hayan introducido nuevos problemas o hayan afectado áreas no relacionadas. Esta prueba abarca no solo el área afectada por el cambio, sino también otras áreas del sistema que podrían haber sido impactadas indirectamente. El objetivo es asegurar que el software en su totalidad siga funcionando correctamente después de los cambios.

Ambos tipos de pruebas son importantes para asegurar la calidad del software y garantizar que los cambios implementados no hayan generado efectos adversos en otras partes del sistema. La realización de estas pruebas ayuda a mitigar el riesgo de introducir nuevos defectos o de regresiones en el software y contribuye a mantener la estabilidad y la funcionalidad del sistema.

### Pruebas de Confirmación - Retesting

La prueba de confirmación, también conocida como retesting, es un tipo de prueba que se realiza después de que se ha detectado y solucionado un defecto en el software. El objetivo principal de esta prueba es confirmar que el defecto original ha sido corregido de manera adecuada y que el software funciona correctamente después de la solución.

Es importante tener en cuenta que la actividad de solucionar un defecto, conocida como debugging, es responsabilidad del equipo de desarrollo y no del equipo de testing. Una vez que el equipo de desarrollo ha realizado la corrección, el equipo de testing debe llevar a cabo la prueba de confirmación para verificar que el defecto ha sido efectivamente solucionado.

Durante la prueba de confirmación, se ejecutan los casos de prueba relevantes que estaban asociados con el defecto solucionado. Se verifica si el software se comporta de acuerdo a lo esperado y si el problema reportado ha sido resuelto satisfactoriamente. Si la prueba de confirmación es exitosa, se considera que el defecto ha sido corregido de manera adecuada.

Es importante realizar la prueba de confirmación de manera rigurosa para asegurarse de que el software funcione correctamente después de solucionar un defecto. Esto ayuda a garantizar la calidad del software y a evitar la reintroducción de problemas previamente resueltos.

### Pruebas de Regresión

La prueba de regresión es un tipo de prueba que consiste en repetir todas las pruebas de un programa después de realizar un cambio, con el objetivo de descubrir cualquier defecto que pueda haber sido introducido o no descubierto como resultado de ese cambio.

El propósito principal de la prueba de regresión es detectar cualquier defecto que pueda haber surgido en el software después de realizar modificaciones, ya sea en el software recientemente testeado, en otro software relacionado o no relacionado al cambio, o incluso debido a un cambio en el ambiente en el que se ejecuta el software.

La prueba de regresión se basa en el riesgo de no encontrar defectos en el software que funcionaba correctamente anteriormente. Aunque el cambio realizado puede parecer trivial, siempre existe la posibilidad de que introduzca nuevos problemas o afecte de alguna manera el funcionamiento previo del software.

Durante la prueba de regresión, se ejecutan nuevamente todas las pruebas que se realizaron anteriormente para verificar que el software sigue funcionando correctamente después del cambio. Esto incluye ejecutar casos de prueba, comprobar la integridad de los resultados y asegurarse de que no haya efectos secundarios no deseados.

La prueba de regresión es fundamental para asegurar la calidad del software y garantizar que los cambios realizados no tengan un impacto negativo en el funcionamiento del sistema. Se recomienda realizarla de manera exhaustiva y sistemática para detectar cualquier problema introducido como resultado de los cambios realizados.

- La prueba de regresión puede realizarse en todos los niveles de prueba, incluyendo pruebas funcionales, pruebas no funcionales y pruebas estructurales. Consiste en ejecutar los casos de prueba nuevamente para verificar que el software sigue funcionando correctamente después de realizar cambios.

- La regresión implica repetir los casos de prueba tantas veces como sea necesario para garantizar la estabilidad del sistema. Debido a la cantidad de pruebas que pueden estar involucradas, la regresión es un fuerte candidato para la automatización.

- Automatizar los casos de prueba de regresión permite ejecutarlos de manera rápida y eficiente, lo que ahorra tiempo y recursos. Además, ayuda a asegurar que se realicen las pruebas de regresión de manera consistente y precisa.

- Al automatizar los casos de prueba de regresión, se puede programar la ejecución de las pruebas en momentos específicos, como después de cada cambio o de forma periódica. Esto asegura que cualquier impacto negativo causado por los cambios sea detectado de manera oportuna.

- La automatización de pruebas de regresión también proporciona la capacidad de generar informes detallados sobre el estado de las pruebas, identificar rápidamente los defectos introducidos y facilitar la colaboración entre los miembros del equipo de prueba.

En resumen, la automatización de los casos de prueba de regresión es altamente recomendada debido a la cantidad de pruebas involucradas y el tiempo que puede requerir su ejecución. Proporciona eficiencia, consistencia y precisión en la ejecución de las pruebas, lo que contribuye a mantener la calidad y estabilidad del software después de realizar cambios.

**Estrategia de pruebas de regresión:**

1. Repetición total de casos de prueba: La estrategia implica ejecutar todos los casos de prueba diseñados para una aplicación a medida que se avanza en la prueba de nuevos módulos o funcionalidades. Esto asegura que las modificaciones realizadas no hayan introducido nuevos errores o afectado negativamente las áreas que anteriormente funcionaban correctamente.

2. Reconocimiento de errores indirectos: La estrategia reconoce que la corrección de errores en una aplicación puede dar lugar a la aparición de errores indirectos en áreas que se suponían libres de problemas. Por lo tanto, se realizan pruebas adicionales y se ponen a prueba las áreas que podrían verse afectadas por los cambios realizados, incluso si no están directamente relacionadas con el área modificada.

La estrategia de pruebas de regresión busca garantizar la estabilidad y la calidad del software a medida que se introducen cambios y mejoras en la aplicación. La repetición de los casos de prueba y la detección de errores indirectos permiten identificar y corregir cualquier impacto negativo que pueda surgir durante el proceso de desarrollo y mantenimiento del software.

**Selección de casos de regresión:**

- Casos reusables: Estos casos corresponden a partes del software que no han sido modificadas, tanto en la especificación como en la implementación. Estos casos de prueba pueden ser reutilizados para verificar que las áreas del sistema que no han sufrido cambios sigan funcionando correctamente.

- Retesteables: Estos casos de prueba se refieren a situaciones en las que la especificación del software no ha cambiado, pero sí ha habido modificaciones en la implementación. Estos casos deben ser reevaluados y ejecutados nuevamente para asegurarse de que los cambios realizados no hayan introducido nuevos errores o afectado negativamente el comportamiento esperado del sistema.

- Obsoletos: Estos son casos de prueba que ya no son relevantes debido a cambios en la especificación, implementación o requisitos del software. Estos casos ya no se pueden utilizar y deben ser descartados o actualizados según corresponda.

Un ejemplo de selección de casos de regresión sería un caso de prueba que especifica una relación 1/0 incorrecta debido a una modificación en la especificación del software. Este caso debe ser revisado y actualizado para reflejar la nueva especificación y luego ser ejecutado para confirmar que el comportamiento esperado se mantiene correctamente.

### ¿Cuándo hacer test de regresión?

El test de regresión se debe realizar en los siguientes momentos:

Durante el desarrollo de software:
- Después de realizar cambios en el código fuente.
- Después de añadir nuevas funcionalidades.
- Después de corregir errores o defectos.

En tareas de mantenimiento:
- Después de aplicar actualizaciones o parches.
- Después de realizar modificaciones en el sistema.
- Después de realizar migraciones de datos.
- Después de realizar cambios en el entorno de desarrollo o producción.

Es importante realizar pruebas de regresión en estos momentos para asegurarse de que los cambios realizados no hayan introducido nuevos errores o afectado negativamente el funcionamiento previamente probado del software.

### Tipos de Regresiones

1. Pruebas Intake
Las pruebas intake o de admisión son un tipo especial de pruebas de humo con el objetivo de determinar si un componente o sistema está en buenas condiciones para realizar pruebas detalladas y continuar con el proceso de regresiones. Estas pruebas se realizan al comienzo de la fase de prueba y su tiempo de ejecución es corto, generalmente no supera los 5 minutos en ejecución automática o 15 minutos en ejecución manual.

2. Pruebas Smoke
Las pruebas smoke son las primeras pruebas de regresión que se realizan en la versión nueva de software, con el objetivo de determinar si los componentes ya construidos están en buenas condiciones para realizar las demás pruebas y continuar con el proceso de regresiones planificado. Estas pruebas suelen representar aproximadamente el 10% del tiempo total de las pruebas completas de regresión.

3. Pruebas por Funcionalidad
Las pruebas por funcionalidad son pruebas de regresión que se realizan en la versión nueva de software para determinar si una funcionalidad ha sido afectada por cambios introducidos indirectamente. Estas pruebas pueden realizarse en dos situaciones: cuando una funcionalidad nueva está por salir a producción y debe ser probada en un entorno integrado, y cuando una funcionalidad existente debe seguir funcionando correctamente a pesar de los cambios realizados en el sistema. El tiempo requerido para estas pruebas depende de la complejidad de la funcionalidad y se utilizan especialmente en proyectos automatizados o en equipos de testing numerosos, donde se pueden dividir las funcionalidades para una mejor gestión.

4. Pruebas de Regresiones Completas
Las pruebas de regresiones completas consisten en la ejecución de todos los casos de pruebas activos que se han ejecutado al menos una vez en algún ciclo de prueba. Estas pruebas se realizan en la versión nueva de software con el objetivo de determinar si se han introducido defectos indirectamente en el sistema desarrollado y que está por salir a producción. Las pruebas de regresiones completas abarcan la totalidad de los casos de prueba y garantizan una cobertura exhaustiva del software.

Recuerda que la selección del tipo de regresión a utilizar dependerá de las características y necesidades del proyecto, así como de los recursos y tiempos disponibles para la ejecución de las pruebas de regresión.

**Regresiones**

1. Las ejecuciones de pruebas de regresión son costosas, ya que implican la repetición de casos de prueba en el sistema completo para detectar cualquier defecto introducido o no descubierto como resultado de los cambios realizados.

2. A medida que el sistema crece, el tiempo necesario para ejecutar las pruebas de regresión también se incrementa, ya que la cantidad de casos de prueba aumenta y los tiempos de ejecución se prolongan.

3. Las necesidades de lanzamiento del software a producción suelen exigir tiempos de regresión cortos pero con una alta calidad de controles. Es importante encontrar un equilibrio entre la velocidad de ejecución y la exhaustividad de las pruebas.

4. La realización de pruebas de regresión es fundamental para garantizar la estabilidad y calidad del software en producción. Sin realizar estas pruebas, se corre el riesgo de tener defectos no detectados que pueden afectar negativamente el funcionamiento del sistema.

5. Debido a la naturaleza repetitiva de las pruebas de regresión y a su importancia crítica, estas pruebas son fuertemente candidatas a ser automatizadas. La automatización de las pruebas de regresión permite reducir los tiempos de ejecución, aumentar la cobertura de pruebas y mejorar la eficiencia del proceso de regresión.
